1 Model (class database)
Trong dự án LaVie: Vì bạn dùng pyodbc và gọi Store Procedure (truy vấn thuần), thư mục models/ của bạn thực chất sẽ chứa các Class đại diện cho dữ liệu trả về từ SQL để Python hiểu được.
Ví dụ: Một Model SignedDocument sẽ có đầy đủ tất cả các trường snapshot và chữ ký để lưu xuống ổ cứng.
2. Schema (validate data) Nhiệm vụ:
Validate (Kiểm tra): Nếu nhân viên FO gửi lên một chuỗi chữ ký bị trống, Schema sẽ báo lỗi ngay lập tức trước khi nó kịp chạm vào Database.
Filter (Lọc): Bạn có 100 cột trong Database, nhưng bạn chỉ muốn trả về cho Lễ tân 5 cột (Tên, Phòng, Mã đơn). Schema giúp bạn "giấu" các thông tin nhạy cảm đi.
Ví dụ: Schema DocumentCreate sẽ không có trường CreatedAt (vì database tự sinh), nhưng Schema DocumentOut sẽ có CreatedAt để hiện lên màn hình CoreUI.
 Tại sao bạn cần tách biệt cả hai? Model Schemas
Để LaVie Project không bao giờ bị lỗi dữ liệu, bạn sẽ dùng luồng này:
Client gửi dữ liệu lên -> Schema kiểm tra (Ví dụ: Tuổi phải là số, Tên không được để trống).
Dữ liệu "sạch" từ Schema được đưa vào Controller.
Controller gọi Service để map dữ liệu đó vào Model (thêm các trường tự động như UpdateAt, UpdateBy).
Model được đẩy xuống SQL Server qua pyodbc.
